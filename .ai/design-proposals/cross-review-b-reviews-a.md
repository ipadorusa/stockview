# Cross Review: Designer B가 Proposal A(Visual System)를 리뷰

> **리뷰어**: UX Designer B (Page Layout Designer)
> **대상**: Proposal A — "Terminal Elegance" Visual System
> **날짜**: 2026-03-29
> **관점**: 페이지 레이아웃 설계자 입장에서 비주얼 시스템이 대시보드 레이아웃을 얼마나 잘 지원하는가

---

## 1. 전체 평가

Proposal A는 매우 체계적이고 완성도 높은 비주얼 시스템이다. 5단계 배경 계층, oklch 기반 색상 과학, 글래스모피즘 카드, 금융 특화 타이포그래피까지 — 금융 대시보드에 필요한 거의 모든 시각적 기반이 갖춰져 있다. Proposal B의 레이아웃들과 높은 호환성을 보인다.

**종합 점수: 9/10** — 레이아웃 관점에서 거의 완벽한 토큰 시스템. 아래 기술하는 몇 가지 보완점만 추가되면 통합 구현에 문제가 없다.

## 2. 디자인 토큰 — 레이아웃 적합성 분석

### 2.1 배경 계층 (5 Level) — 매우 우수

Proposal B의 레이아웃은 여러 깊이의 패널이 중첩되는 구조다:

| Proposal B 컴포넌트 | 적합한 A 배경 레벨 | 적합도 |
|---|---|---|
| 페이지 전체 배경 | L0 `--bg-base` | 완벽 |
| 티커 테이프 바 | L1 `--bg-surface` | 완벽 |
| 인덱스 위젯 카드 | L2 `--bg-card` + 방향성 그래디언트 | 완벽 |
| 사이드바 패널 (종목 상세) | L1 `--bg-surface` | 완벽 |
| 사이드바 내 통계 카드 | L2 `--bg-card` | 완벽 |
| 툴팁, 차트 크로스헤어 | L4 `--bg-floating` | 완벽 |
| 헤더 (glass) | `--glass-bg` | 완벽 |
| 모바일 바텀시트 | L3 `--bg-elevated` | 완벽 |

**판정**: 5단계 계층이 Proposal B의 모든 레이아웃 깊이를 커버한다. 추가 레벨 불필요.

### 2.2 텍스트 계층 (4 Level) — 우수

| Proposal B 용도 | 적합한 A 텍스트 토큰 | 비고 |
|---|---|---|
| 대형 가격 표시 (4xl) | `--text-primary` | 정확 |
| 종목명, 섹션 제목 | `--text-primary` | 정확 |
| 카드 라벨 (KOSPI, PER 등) | `--text-secondary` | 정확 |
| 타임스탬프, 출처 | `--text-tertiary` | 정확 |
| 플레이스홀더, 비활성 | `--text-muted` | 정확 |

**판정**: 4단계 텍스트 계층이 데이터 밀도 높은 대시보드의 모든 텍스트 역할을 충분히 구분한다.

### 2.3 주식 색상 — 호환성 확인 필요

Proposal A는 주식 색상을 oklch로 재정의했다:
- Up: `oklch(0.55 0.22 25)` (light) / `oklch(0.72 0.20 25)` (dark)
- Down: `oklch(0.55 0.17 250)` (light) / `oklch(0.72 0.15 250)` (dark)

현재 코드베이스는 hex 값 사용:
- Up: `#e53e3e` (light) / `#fc8181` (dark)
- Down: `#3182ce` (light) / `#63b3ed` (dark)

**권장사항**: oklch 전환은 시각적으로 유사한 결과를 줄 수 있지만, 기존 hex 값과의 시각적 동등성을 반드시 검증해야 한다. 특히 `--color-stock-up`은 한국 투자자에게 "빨강=상승"이라는 강한 인식이 있으므로, oklch 변환 후에도 충분히 "빨간색"으로 인식되는지 확인 필요.

## 3. 카드 변형 — 레이아웃별 매핑

Proposal A가 제공하는 5가지 카드 변형이 Proposal B의 어떤 컴포넌트에 쓰일지 구체적으로 매핑한다.

### 3.1 매핑 테이블

| Proposal B 컴포넌트 | A 카드 변형 | 이유 |
|---|---|---|
| **홈 — 인덱스 위젯 (KOSPI 등)** | `card-stat` | 방향성 좌측 보더 + 그래디언트 배경이 정확히 내가 설계한 인덱스 카드와 일치 |
| **홈 — 뉴스 피드 카드** | `card-interactive` | 클릭 가능, 호버 시 리프트 효과 필요 |
| **홈 — 퀵링크 카드** | `card-interactive` | 동일한 이유 |
| **홈 — 센티먼트 게이지 패널** | `card-default` | 정적 디스플레이, 인터랙션 없음 |
| **홈 — 환율 카드** | `card-stat` | 인덱스 위젯과 동일 패턴 |
| **시장 — 섹터 히트맵 컨테이너** | `card-default` | 내부에 히트맵 그리드가 들어가는 외부 컨테이너 |
| **시장 — 상승/하락 모멘텀 바** | `card-default` | 정적 데이터 디스플레이 |
| **시장 — 정렬 가능 테이블** | `card-default` | 테이블 감싸는 컨테이너 |
| **종목 상세 — 사이드바 통계** | `card-stat` | 핵심 지표 표시에 최적 |
| **종목 상세 — 관련 종목** | `card-interactive` | 클릭하여 종목 이동 |
| **종목 상세 — 52주 레인지** | `card-default` | 시각화 컨테이너 |
| **헤더** | `card-glass` (header-glass) | 글래스모피즘 헤더에 완벽 |
| **모바일 바텀탭 바** | `card-glass` 변형 | frosted glass 효과와 일치 |

### 3.2 누락된 카드 변형

**`card-ticker`** — 티커 테이프 전용 카드가 없다. Proposal B의 티커 테이프 바는 독특한 레이아웃 요소로, 기존 카드 변형 어디에도 정확히 맞지 않는다. `card-glass`의 변형으로 처리하거나, 별도 유틸리티 스타일로 정의하는 것을 권장한다.

**`card-chart`** — 차트 영역을 감싸는 컨테이너 스타일이 명시적으로 없다. 차트 영역은 테두리 처리, 내부 패딩, 도구 모음과의 관계가 다른 카드들과 다르다. `card-default`의 변형으로 `padding: 0`, `overflow: hidden`을 적용한 차트 전용 변형이 필요하다.

## 4. 타이포그래피 스케일 — 데이터 밀도 적합성

### 4.1 매우 우수한 점

- **`--text-2xs` (10px)** 추가: Proposal B의 차트 축 라벨, 거래량 표시에 정확히 필요한 크기. 현재 Tailwind 기본에는 없는 크기라 이 추가가 매우 유용하다.
- **`--text-sm` (13px)**: 기존 Tailwind `text-sm`이 14px인데, 13px로 한 단계 더 촘촘하게 정의한 것이 데이터 밀도 높은 테이블에 적합하다.
- **`--text-md` (15px)**: base(14px)와 lg(17px) 사이의 갭을 메우는 사이즈. 카드 제목에 유용.
- **`font-price` 유틸리티**: `tabular-nums` + 볼드 + 타이트한 `letter-spacing`이 금융 대시보드에 필수적. Proposal B의 모든 가격 표시에 사용할 수 있다.

### 4.2 보완이 필요한 점

**대형 디스플레이 숫자 부족**: Proposal B의 종목 상세 페이지에서 메인 가격은 `text-4xl` (36px)을 사용하는데, 이는 A의 스케일 최대치다. 홈페이지 히어로나 랜딩 페이지에서 더 큰 숫자가 필요할 수 있으므로 `--text-5xl` (48px) 정도를 하나 더 추가하는 것을 제안한다. 다만 현재 설계에서는 36px이면 충분하므로 우선순위는 낮다.

**라인하이트 최적화**: Proposal A의 큰 사이즈(2xl~4xl)에 `line-height: 1.1~1.2`를 적용한 것이 좋다. 대형 가격 숫자가 카드 내에서 불필요한 수직 공간을 차지하지 않게 해준다. Proposal B의 인덱스 카드(높이 140px)에서 이 타이트한 라인하이트가 공간 효율에 직접 기여한다.

## 5. 마이크로인터랙션 — 레이아웃 인터랙션과의 정합성

### 5.1 잘 맞는 부분

| A의 인터랙션 | B의 레이아웃 요소 | 정합도 |
|---|---|---|
| `price-tick-up/down` 애니메이션 | 티커 테이프, 인덱스 카드, 종목 행 가격 | 완벽 — B에서 설계한 "가격 변동 플래시"와 정확히 동일한 의도 |
| `card-interactive` 호버 (translateY -2px) | 뉴스 카드, 퀵링크, 인기 종목 행 | 완벽 — B에서 `hover:shadow-md hover:-translate-y-0.5`와 동일 |
| `animate-count` (숫자 카운트업) | 인덱스 위젯 초기 로딩 | 완벽 — B에서 설계한 "0에서 실제 값으로 600ms 카운트업"과 일치 |
| `tooltip-enter` 애니메이션 | 차트 크로스헤어 툴팁 | 완벽 — glass 배경 + 페이드인 |
| `page-enter` 전환 | 모든 페이지 콘텐츠 | 완벽 — B에서 설계한 "콘텐츠 페이드인 200ms"와 호환 |
| `skeleton-shimmer` | 모든 로딩 상태 | 우수 — B의 "프로그레시브 스켈레톤"과 호환, gradient sweep이 기존 pulse보다 고급스러움 |
| `star-pop` 애니메이션 | 종목 상세 관심종목 버튼 | 완벽 |

### 5.2 누락된 인터랙션

**1. 티커 테이프 스크롤 애니메이션**: Proposal B에서 설계한 `@keyframes ticker`(무한 수평 스크롤)에 대한 정의가 A에 없다. A의 `--duration-*`과 `--ease-*` 토큰은 단발성 전환용이지 무한 루프 애니메이션용이 아니다. 티커 테이프 전용 CSS 애니메이션을 A 시스템에 추가해야 한다.

**2. 탭 전환 애니메이션**: B에서 설계한 탭 인디케이터의 "슬라이드 이동" (`transition: left 200ms, width 200ms`) 전용 정의가 없다. A의 일반적인 `--duration-normal` + `--ease-out`을 사용하면 되지만, 명시적으로 탭 인디케이터 패턴을 문서화하면 좋겠다.

**3. 캐러셀 스크롤 스냅**: 모바일 인덱스 카드 캐러셀의 `scroll-snap` 관련 스타일이 없다. 이는 CSS 네이티브 기능이므로 A 시스템에 반드시 포함될 필요는 없지만, 스크롤 인디케이터(dots)의 스타일은 정의되면 좋겠다.

**4. 차트 로딩 — 라인 그리기 애니메이션**: B에서 설계한 "스파크라인이 좌→우로 그려지는 600ms 애니메이션"에 대한 정의가 없다. lightweight-charts 라이브러리 내부에서 처리할 수도 있지만, CSS 기반 미니 차트용 그리기 애니메이션이 있으면 유용하다.

## 6. 데이터 시각화 색상 — 레이아웃 특화 분석

### 6.1 히트맵 그래디언트 — 완벽

A의 9단계 히트맵 그래디언트(`--heatmap-1` ~ `--heatmap-9`)가 B의 섹터 트리맵에 정확히 매핑된다:

```
B의 섹터 히트맵 셀:
  강한 상승(>2%): --heatmap-8 or --heatmap-9
  상승(0~2%):     --heatmap-6 or --heatmap-7
  보합:           --heatmap-5
  하락(0~-2%):    --heatmap-3 or --heatmap-4
  강한 하락(<-2%): --heatmap-1 or --heatmap-2
```

B에서 단순히 "red-500, red-300, gray-400, blue-300, blue-500"으로 정의했던 것보다 A의 oklch 기반 9단계가 훨씬 세밀하고 일관성 있다. **A의 히트맵 색상을 그대로 채택해야 한다.**

### 6.2 마켓 인덱스 색상 — 우수한 추가

A가 `--index-kospi`, `--index-kosdaq`, `--index-sp500`, `--index-nasdaq`, `--index-usdkrw` 전용 색상을 정의한 것이 좋다. B의 인덱스 위젯 카드에서 각 지수를 시각적으로 구분하는 데 활용할 수 있다. B에서는 방향성(up/down) 색상만 사용했는데, A의 인덱스별 고유 색상을 미니 차트 라인 색상으로 쓰면 더 풍부한 시각적 구분이 가능하다.

### 6.3 뱃지/카테고리 색상 — 완벽

A의 `Market KR`, `Market US`, `Sector`, `News`, `ETF` 뱃지 색상이 B의 뉴스 카드 카테고리 뱃지, 시장 태그, 섹터 표시에 직접 매핑된다. B에서 "경제(blue), 기업(green), 증시(amber), 해외(purple)"로 임의 지정했던 것을 A의 체계적 뱃지 시스템으로 통일해야 한다.

### 6.4 누락 — 마켓 브레드스 바 색상

B의 "마켓 브레드스 바" (상승:보합:하락 비율을 보여주는 수평 바)에 전용 색상이 없다. `--color-stock-up`, `--color-stock-flat`, `--color-stock-down`을 그대로 사용할 수 있지만, 바 형태에서는 투명도 조절이 필요할 수 있다. `--color-stock-up` at 70% 같은 중간 투명도 토큰이 있으면 편리하다.

## 7. 글래스모피즘 — 레이아웃 활용 포인트

A의 글래스모피즘 시스템이 B의 레이아웃에서 활용될 수 있는 곳:

| 위치 | 사용 방식 | 효과 |
|---|---|---|
| **헤더** | `header-glass` — A가 이미 정의 | 스크롤 시 콘텐츠가 비쳐보이는 고급감 |
| **모바일 바텀탭** | `--glass-bg` + `backdrop-blur` | B에서 설계한 "frosted glass" 효과와 정확히 일치 |
| **차트 툴팁** | `chart-tooltip` — A가 이미 정의 | 차트 위 데이터 표시에 최적 |
| **모바일 스티키 CTA** | `--glass-bg` 변형 | 종목 상세의 "관심종목 추가" 플로팅 버튼 |
| **티커 테이프** | 사용하지 않음 | 티커 테이프는 불투명 배경이 가독성에 유리 |

**주의사항**: `backdrop-filter: blur()`는 성능 비용이 있다. 특히 모바일에서 여러 글래스 요소가 동시에 렌더링되면 프레임 드롭이 발생할 수 있다. B의 레이아웃에서 글래스 효과는 **헤더 + 바텀탭**으로 제한하고, 나머지는 솔리드 배경을 권장한다.

## 8. 통합 권장사항

### 8.1 즉시 채택 (변경 없이 그대로 사용)

1. **배경 5단계 계층** — B의 모든 레이아웃 깊이를 완벽하게 지원
2. **카드 시스템 5종** — `card-stat`이 인덱스 위젯에, `card-interactive`가 목록 항목에 정확히 매핑
3. **히트맵 9단계 그래디언트** — B의 섹터 트리맵에 그대로 적용
4. **마켓 인덱스 전용 색상** — 인덱스 위젯 미니 차트 라인 색상으로 활용
5. **뱃지 색상 체계** — B의 임의 색상 대신 A의 체계적 뱃지 시스템 사용
6. **타이포그래피 스케일** — 특히 `text-2xs`, `text-md`, `font-price` 유틸리티
7. **마이크로인터랙션 대부분** — price-tick, card hover, tooltip enter, skeleton shimmer
8. **shadow 계층 4단계** — B의 hover 효과에 직접 매핑
9. **이징 함수 체계** — `--ease-out`, `--ease-spring` 등 B의 모든 전환에 활용
10. **접근성 검증 결과** — WCAG AA/AAA 준수 확인 완료

### 8.2 A에 추가 요청

1. **티커 테이프 스타일 정의**: 무한 스크롤 애니메이션, 배경 스타일, 호버 시 일시정지
2. **차트 컨테이너 카드 변형** (`card-chart`): `padding: 0`, `overflow: hidden`, 도구 모음 영역 분리
3. **스크롤 인디케이터(dots) 스타일**: 모바일 캐러셀 페이지네이션 도트의 active/inactive 색상
4. **마켓 브레드스 바 투명도 토큰**: 수평 바 차트에서 주식 색상의 70% 투명도 변형

### 8.3 B에서 A에 맞춰 수정할 부분

1. **색상 값 통일**: B에서 hex로 정의한 모든 색상(`#e53e3e`, `#3182ce` 등)을 A의 oklch 값으로 교체
2. **카드 스타일 통일**: B에서 개별 정의한 카드 hover/shadow를 A의 `card-interactive`, `card-stat` 시스템으로 교체
3. **뉴스 카테고리 뱃지**: B의 임의 색상(blue, green, amber, purple) → A의 뱃지 색상 체계로 교체
4. **스켈레톤 로딩**: B의 `animate-pulse` → A의 `skeleton-shimmer` (gradient sweep)로 업그레이드
5. **히트맵 색상**: B의 Tailwind 색상 클래스(red-500 등) → A의 `--heatmap-*` CSS 변수로 교체

### 8.4 공동 결정이 필요한 사항

1. **다크 모드 우선 vs 라이트 모드 우선**: A는 "다크 모드가 primary experience"라고 명시. B는 양쪽을 동등하게 설계. 사용자 대상(초보 투자자)을 고려하면 라이트 모드 기본, 다크 모드 옵션이 더 접근성이 높을 수 있다. 사용자 테스트로 결정 필요.

2. **글래스모피즘 범위**: A는 적극적 사용(헤더, 카드, 툴팁), B는 제한적 사용(헤더 + 바텀탭만) 권장. 성능과 시각적 효과의 균형점을 구현 단계에서 테스트해야 한다.

3. **노이즈 텍스처 오버레이**: A가 제안한 `texture-noise`는 시각적으로 고급스럽지만, 데이터 밀도가 높은 대시보드에서는 가독성을 해칠 수 있다. 빈 상태(empty state)나 히어로 섹션에만 제한적으로 사용하고, 데이터 영역에서는 제외하는 것을 권장한다.

## 9. 결론

Proposal A의 비주얼 시스템은 Proposal B의 페이지 레이아웃을 구현하기에 **매우 적합**하다. 특히 다음 세 가지가 두 제안의 시너지 포인트다:

1. **`card-stat` + 인덱스 위젯**: A의 stat 카드가 방향성 좌측 보더 + 그래디언트 배경을 이미 정의하고 있어, B의 인덱스 위젯 카드를 코드 한 줄의 수정 없이 구현 가능
2. **히트맵 그래디언트 + 섹터 트리맵**: A의 9단계 oklch 히트맵이 B의 섹터 트리맵에 정확히 매핑
3. **글래스모피즘 + 헤더/바텀탭**: A의 glass 토큰이 B의 네비게이션 재설계에 완벽하게 맞음

보완이 필요한 4가지 항목(8.2절)은 비교적 작은 추가 작업이며, B 측에서 수정할 5가지 항목(8.3절)도 기계적 교체 수준이다. 두 제안의 통합은 충돌 없이 순조롭게 진행될 것으로 판단한다.
